## 3단계 탐색 프로토콜

비용 및 토큰 최적화를 위해 다음 3단계 계층을 엄격히 준수해야 합니다. **본문 전역 검색을 즉각적으로 실행하는 것은 금지됩니다.**

1. **1단계: 인덱스 스캔 (최저 비용)**
   - 질문이나 작업과 연관된 도메인의 최상위 인덱스 파일(예: `30_index.md`) **단 1개만** 로드하여 기본 구조를 파악합니다.
   - 디렉토리 통째 크롤링은 금지됩니다. 하위 인덱스를 선별 로드합니다.

2. **2단계: 자율 경로 탐색 (비용 최적화)**
   - 1단계(인덱스 스캔) 결과로 정보가 충분하지 않은 경우, 본문 검색으로 넘어가기 전 **디렉토리/파일명 기반 탐색**을 수행합니다.
   - `list_directory`, `glob` 도구 등을 활용하여 관련 프로젝트나 하위 폴더(`예: 회의록/`)의 파일명 목록을 스캔합니다.
   - 파일명만으로 유력한 후보를 찾으면, 해당 파일 단 1개만 열어 내용을 확인합니다.

3. **3단계: 전역 본문 검색 (최후 수단, 최고 비용)**
   - 1, 2단계를 통해서도 목적을 달성할 수 없는 경우에 한해, 아래 "검색 도구 계층"에 명시된 본문 검색을 실행합니다.

## 검색 도구 계층 (3단계 전역 본문 검색용)

*(이 절은 index.md 도구 활용 대원칙을 검색 맥락에서 구체화한 계층입니다. 최후 수단으로만 사용하십시오.)*

1. **1순위:** `obsidian search <키워드>` (Obsidian 플러그인 활성 시 우선 활용)
2. **2순위:** `grep_search` 등 전역 텍스트 검색 도구 (검색어 최소화, 대상 범위 좁혀서 사용. **옵시디언 볼트 루트에서 직접 광범위한 검색 실행은 절대 금지**)

## 장기 인덱스 확장 토큰 최적화

단일 인덱스가 100줄을 넘어갈 경우 (`22_index.md`처럼) 하위 카테고리 트리로 2차 분산을 검토 권고합니다.
새로운 도메인 디렉토리 생성 시 반드시 최소 규격의 빈 인덱스 파일(예: `21_index.md`)을 자동 생성하여 탐색 실패를 방지합니다.

## 크로스도메인 토큰 낭비 제어

2개 이상의 도메인 동시 로드는 원칙상 낭비입니다.  
질문의 **'행위자' 혹은 '동사/주요 액션'이 속한 도메인 인덱스를 1순위로 단독 로드**합니다.  
정보 부족 시 다음 단계(2단계 자율 경로 탐색 -> 3단계 전역 검색 도구 우회)로 진행합니다.
