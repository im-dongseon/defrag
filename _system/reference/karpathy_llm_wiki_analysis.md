# LLM Wiki (Andrej Karpathy) - 원문 분석 보고서

## 1. 개요
* **출처:** https://gist.github.com/karpathy/442a6bf555914893e9891c11519de94f
* **주제:** 기존의 RAG(Retrieval-Augmented Generation) 방식을 넘어, LLM이 주도하여 영구적이고 누적되는 형태의 개인화된 구조적 위키(Knowledge Base)를 구축하는 패턴

## 2. 핵심 문제의식
* 기존 RAG 시스템의 한계: 문서를 벡터 데이터베이스에 넣고 검색할 때마다 문맥을 매번 새롭게 조립하므로, 지식이 "누적(Compounding)"되지 않으며 복잡한 문서 간의 연결이나 합성을 파악하기 어렵습니다.

## 3. 제안하는 아키텍처 및 원리
지식을 3개의 계층으로 나눕니다.
1. **Raw sources (원천 데이터):** 변경 불가한 텍스트, 이미지, 논문 등의 원본 스토리지
2. **The Wiki (위키 계층):** LLM이 전적으로 관리하고 생성하는 마크다운 파일들의 집합. 개념/엔티티 페이지, 본문 요약, 교차 참조 링크(`[[ ]]`)를 포함.
3. **The Schema (스키마):** LLM이 위키를 관리하는 방법을 규정한 지침 (예: `AGENTS.md`)

## 4. 운영 프로세스 (Operations)
* **Ingest (수집):** 새 원본 문서가 들어오면 LLM이 이를 읽고 요약 페이지 생성, 관련 개념 페이지 갱신, 백링크 연결 및 로그 작성을 일괄 수행.
* **Query (질의):** 질문을 받으면 LLM이 위키의 `index.md`를 먼저 스캔해 관련 페이지를 찾아 읽고, 합성된 답변을 도출하여 그 자체를 다시 위키의 새 페이지로 저장.
* **Lint (점검):** 정기적으로 모순점, 고립된 페이지(Orphan), 오래된 내용 등을 점검해 지식의 무결성 유지.

## 5. 주요 메타데이터 구조
* `index.md`: 위키 내 모든 항목에 대한 카탈로그 (분류, 요약 포함). 중간 규모까지는 별도의 벡터 DB 없이 이 인덱스만으로 효율적인 탐색 가능.
* `log.md`: 시스템에서 발생한 이벤트(수집, 쿼리 등)의 타임라인 기록 (예: `## [2026-04-02] ingest`).

## 6. 결론 및 인사이트
인간은 소스를 큐레이팅하고 질문하는 데 집중하고, 북키핑(교차 링크 관리, 갱신, 요약 등)의 지루한 작업은 LLM에게 위임함으로써, 개인의 지식이 유지보수 비용 없이 '복리'로 성장할 수 있는 지속 가능한 시스템 패턴을 제시합니다.
