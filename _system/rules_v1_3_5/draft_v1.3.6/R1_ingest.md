# R1. 수집(Ingest) 및 파일 포맷별 컴파일 분기 파이프라인

← [index.md](index.md)로 돌아가기

---

모든 Inbox 파일 처리는 포맷에 따라 분기됩니다.  
`00_Inbox/unsorted/` 내 파일은 신규 수집 대상이 아니므로 이 단계에서 건너뜁니다.

## 1단계. 포맷 판별 및 분기

- **텍스트 (.md, .txt 등):** 내용 길이와 관계없이 **즉시 유사도 비교(2단계) 직행**. 50자 미만의 짧은 텍스트라도 바이너리 추출 실패 로직을 적용하지 않습니다.
- **바이너리/미디어 (PDF, 이미지, 오디오, 비디오 등):** 텍스트/표 맞춤 추출 후 3단계 컴파일 직행.  
  **[대상 확장자]:** 
  - **문서:** `.pdf`, `.epub`
  - **이미지:** `.jpg`, `.jpeg`, `.png`, `.gif`, `.webp`, `.svg`, `.heic`, `.tiff`
  - **오디오:** `.m4a`, `.mp3`, `.wav`, `.aac`, `.flac`, `.ogg`
  - **비디오:** `.mp4`, `.mov`, `.avi`, `.mkv`, `.webm`
  **[Sidecar 우선 원칙]:** 텍스트 추출 성공 여부와 관계없이 **반드시 Sidecar(.md) 노트를 먼저 생성한 후 원본을 이동**해야 합니다. 원본 파일이 아카이브로 이동하기 전에는 연결된 Sidecar가 반드시 존재해야 합니다.
  추출에 실패하거나 추출된 내용이 50자 미만일 시 → `status: idea`의 빈 Sidecar 노트를 즉시 생성하고, 원본과의 연결성을 위해 `location` 정보를 남깁니다.
  이후 원본 파일은 `00_Inbox/unsorted/`로 이동한 뒤 Dashboard에 `[Unsorted][YYYYMMDD] 파일명`을 기록합니다.  
  추출 실패 Sidecar에는 **임시값으로** 현재 원본 위치(`00_Inbox/unsorted/파일명`)를 가리키는 **`location`**을 반드시 남깁니다.  
  이 **`location`**은 R4 Lint Step 5에서 원본이 Archive로 이주된 직후 최종 경로로 갱신됩니다.

## 2단계. 유사도 선행 비교 (텍스트 전용)

- 스캔 즉시 Dashboard에 `[Cache][YYYYMMDD] (내용 요약)` 기록.
- **80% 이상 유사 판단:** 에이전트의 재량에 위임하되, 판단 근거(예: "주요 헤딩/키워드 3/4 일치")를 `[Log]`에 1줄 기재. 80% 이상 → 기존 노트에 **Rewrite 통합** (→ [R7_merge.md](R7_merge.md) 위임).
- **80% 미만** → 신규 컴파일(3단계).

## 3단계. 컴파일/Sidecar 노트 생성

- **텍스트:** **`[카테고리번호] 세부_내용_제목_YYYY-MM-DD.md`** 포맷으로 위키에 배치. 배치 위치는 [R6_metadata.md](R6_metadata.md)의 `type` 기준을 따릅니다.
  - **[파일명 중복 배제 원칙 (Deduplication)]**: 파일명에 상위 폴더명(카테고리명, 프로젝트명)을 중복 기재하지 마십시오. 이미 폴더 구조로 맥락이 파악되므로, 파일명은 핵심 주제 중심으로 최소화합니다.
    - **나쁜 예:** `30_Project-X/회의록/30_Project-X_회의록_2026-04-09.md`
    - **좋은 예:** `30_Project-X/회의록/30_회의록_2026-04-09.md`
  - **[동일 날짜 다중 파일 대응 (Uniqueness)]**: 같은 날짜에 여러 문서(회의록 등)가 발생할 경우, 파일명 뒤에 `_차수` 또는 `_세부주제` 접미사로 추가하여 충돌을 방지하십시오.
    - **예시:** `30_회의록_2026-04-09_2.md` 또는 `30_회의록_2026-04-09_주간공유.md`
  - **[전역 디렉토리 명칭 규칙 (Johnny.Decimal)]:** `10-19` (행정), `20-29` (지식/기술), `30-39` (프로젝트) 하위에 생성되는 **메인 폴더(카테고리)**는 반드시 `[할당번호]_[이름]` 형식이어야 합니다.
    - 기존 카테고리/프로젝트인 경우, 기존 번호(예: `20_`, `30_`)를 유지합니다.
    - 신규 카테고리/프로젝트인 경우, 해당 영역(`10-19`, `20-29`, `30-39`) 내 마지막 할당 번호 + 1을 부여합니다. (예: `20_Category-A` 다음은 `21_Category-B`)
  - **[주제 폴더 필수화 및 번호 금지]:** 옵시디언 볼트 루트나 메인 번호 폴더 바로 아래에 파일을 배치하지 않고, 반드시 하위 주제 폴더(예: `참고자료`, `회의록`, `개념`, `가이드` 등)를 생성하여 배치합니다. **이때, 하위 주제 폴더명에는 `[번호]_` 접두사를 붙이지 않습니다.**
  - `type: admin` → `10-19/[분류번호]_[카테고리명]/[번호 없는 세부주제]/`
  - `type: tech` → `20-29/[분류번호]_[카테고리명]/[번호 없는 세부주제]/`
  - `type: project` → `30-39/[분류번호]_[카테고리명]/[번호 없는 세부주제]/`
    - **[주제 식별 우선순위]:** 
      1. **사용자 직접 입력 (최우선):** 대화 중 명시된 문서 주제 사용.
      2. **파일명 및 내용 분석 (2순위):** 키워드(회의록, 리서치, 스펙 등)를 분석하여 주제를 추론.
      3. **Frontmatter (3순위):** `tags`나 `category` 필드 참조.
  - 판단 불명 시 → `00_Inbox/unsorted/`로 임시 이동하고 Dashboard에 `[Unsorted][YYYYMMDD] 파일명` 마킹
  - **바이너리:** 추출 내용을 기반으로 Sidecar 노트 생성.  
  Frontmatter: `type: source`, **`location`** (URL Encoded Obsidian URI).  
  - **포맷:** `obsidian://open?vault=defrag&file=[URL_Encoded_Archive_Path]`
  - **주의:** 본문 최상단에 마크다운 링크를 별도로 삽입하지 않습니다 (Frontmatter의 `location` 필드가 클릭 가능한 링크 역할을 수행함).
  Sidecar 배치: 도메인 판단이 가능한 경우 아래 예시와 같이 `[분류번호]_[이름]` 및 `[세부주제]` 폴더를 포함하여 배치하고,  
  - 행정 예시: `10-19/10_Category-A/주제-A/파일명.md`
  - 기술 예시: `20-29/20_Category-B/주제-B/파일명.md`
  - 프로젝트 예시: `30-39/30_Project-X/참고자료/파일명.md`

  불명확한 경우 `00_Inbox/unsorted/`에 임시 배치 후 Dashboard에 `[Unsorted][YYYYMMDD] 파일명` 마킹.


## 4단계. 완료 판단 및 원본 처리

- **바이너리 원본:** 즉시 `91_Raw_Archive/assets/[도메인]/`으로 이동 아카이빙 (R8 Archive 쓰기 예외 허용 — [R8_security.md](R8_security.md) 참조).  
  - **[아카이브 경로 정규화]:** `assets/` 하위의 `[도메인]` 폴더명은 반드시 위키의 메인 폴더명인 **`[번호]_[이름]`** 형식을 100% 동일하게 따라야 합니다. (예: `30_Project-X` 폴더의 자산은 `91_Raw_Archive/assets/30_Project-X/`에 배치)
  이후 Archive 구역 파일 재이동/탐색 금지.  
  도메인 불명 시 → `00_Inbox/unsorted/`로 이동 + Dashboard에 `[Unsorted][YYYYMMDD] 파일명` 마킹.  
  이 시점에서 아카이빙은 보류되며, R4 Lint Step 5에서 도메인 재판단 후 재배치를 재시도합니다.
  **링크 및 인덱스 갱신 의무:** 아카이빙 완료 직후 Sidecar의 Frontmatter **`location`** 필드를 최종 Archive 경로를 포함한 **Obsidian URI**로 즉시 갱신합니다. 또한, 파일 이동(`mv`) 후에는 반드시 **Obsidian CLI를 사용하여 볼트 인덱스를 갱신**(`obsidian eval`)해야 합니다.
- **텍스트 원본(`.md`, `.txt`):** 위키 이동 및 인덱싱 갱신이 **100% 정상 완료**된 것을 확인한 후, Inbox에서 원본 파일을 영구 삭제(대체) 조치. 삭제 후에는 반드시 볼트 상태를 최신화하십시오.
