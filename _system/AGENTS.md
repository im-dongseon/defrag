# Meta-Instructions for AI Agents

이 파일은 GitHub Copilot, Claude, Codex 등 모든 AI 에이전트가 작업 전 우선적으로 참고해야 할 메타 지시사항(Meta-Instructions)을 담고 있습니다. 별도의 명시적 지시가 없더라도 이 규칙을 기본적으로 따르십시오.

## 0. 기본적인 정책

- 답변은 과장하지 말고, 사실에 기반하여 작성할 것
- 아부 금지
- 한국어 사용

## 현재 활성 Rules 경로

- **최신 버전:** `_system/rules_v1_3_6/`
- **진입점:** `_system/rules_v1_3_6/index.md`

## Superpowers 사용 원칙

이 저장소에서는 superpowers skill을 사고 프레임 및 절차 보조 도구로만 사용합니다.

1. **정본 우선 원칙:** 규칙 관련 설계, 리뷰, 반영 보고서, 계획 문서는 항상 `_system/rules_v*/draft_v*` 및 최종 `rules_v*` 디렉토리를 정본으로 사용합니다.
2. **산출물 경로 고정:** `docs/superpowers/*` 경로는 이 저장소에서 사용하지 않습니다. superpowers가 별도 산출물 경로를 제안하더라도 현재 rule 체계의 정본 경로로 재매핑합니다.
3. **프로젝트 루트 해석:** superpowers 프로젝트 설정이 필요할 경우, 저장소 루트가 아니라 `_system/` 디렉토리를 기준 루트로 간주합니다.
4. **충돌 시 우선순위:** 사용자 지시와 `_system/AGENTS.md`, 현재 활성 rules가 superpowers 기본 워크플로우보다 우선합니다.


## 1. 시스템 룰 파일의 버전 관리 정책 (Rule File Versioning Policy)

시스템 규칙을 배포하고 업데이트할 때 아래의 프로세스와 컨벤션을 엄격히 준수합니다.

### 1-1. 기본 네이밍 컨벤션
- **파일/디렉토리명:** 규칙 버전 디렉토리에는 `rules` (복수형) 네이밍을 사용합니다. (예: `rules_v1_1` 디렉토리)
- **기준 참조 경로:** 규칙 문서의 참조 기준점은 AGENTS.md가 안내하는 최신 위치를 따릅니다. 특정 루트 심볼릭 링크의 존재를 전제로 문서를 작성하지 않습니다.

### 1-2. 버전 관리 및 업데이트 프로세스 (Draft 단계)
1. **버전 번호 규칙:**
   - 버전은 `Major.Minor.Patch` 형식을 따릅니다.
   - 각 자리수(Minor, Patch 등)는 **9를 넘어 10, 11 등 다자릿수로 계속 증가할 수 있습니다.** (예: `v1.2.9` -> `v1.2.10`)
   - **상위 자리수 승격:** 큰 기능 추가나 구조적 변경이 있을 때에만 상위 자리수를 올립니다. (단순히 9 다음이라서 올리지 않습니다.)
2. **작업 디렉토리 마련:** 업데이트 작업을 시작할 때, 현재 규칙 디렉토리 하위에 다음 버전을 위한 `draft_v{다음버전}` 디렉토리를 생성합니다. 
   - 예시: `rules_v1_3_0/draft_v1_3_1/` 또는 `rules_v1_3_0/draft_v1.4.0/`
3. **리뷰 문서 생성 및 검토:** `draft` 디렉토리 내에 리뷰 문서(`review_v{다음버전}.md`)를 만들어 사용자의 피드백 및 요구사항을 취합합니다.
4. **요구사항 반영 및 리포트:** 취합된 리뷰 문서를 기반으로 룰 초안에 반영하고, 미반영된 사항이나 결함이 있다면 `review_applied_report_v{다음버전}.md`를 작성하여 보고합니다.
5. **미반영 사항 재검토 (Iteration):**
   - 검토 후 추가 논의가 필요한 미반영 사항이 있을 경우, 기존 리뷰 문서의 하단에 신규 섹션을 추가하여 정리합니다.
   - **[중요]** 이력이 섞이지 않도록 신규 섹션을 추가할 때는 **`## [Iter-2] 미반영 재검토 (YYYY-MM-DD)`** 와 같이 명확한 회차 번호나 날짜 헤더를 반드시 달아주어야 합니다.
   - 미반영 사항이 모두 해소될 때까지 이 과정을 반복합니다.

### 1-3. 최종 버전 릴리즈 및 아카이빙
- **최종 저장:** 미반영 사항이 정리되고 업데이트가 확정되면, 최종 룰과 관련 파일들은 새롭게 만들어진 신규 버전 디렉토리(예: 최상위 `rules_v1_2/`)에 분리하여 저장합니다.
- **심볼릭 링크 갱신 (필수):** 신규 버전 디렉토리를 최종 저장한 즉시, `_system/rules` 심볼릭 링크를 해당 최신 `rules_v*` 디렉토리를 가리키도록 갱신한다. 이 단계를 완료하기 전까지는 릴리즈를 완료된 것으로 간주하지 않는다.
- **정합성 검증 (필수):** 링크 갱신 후, `_system/AGENTS.md`의 최신 버전 표기 및 진입점 경로가 실제 릴리즈된 버전 디렉토리 및 심볼릭 링크 대상과 일치하는지 반드시 확인한다. 불일치 발견 시 즉시 수정 후 릴리즈를 확정한다.
- **산출물 보존:** 리뷰 진행, 반영 내역 등 업데이트를 위해 발생한 도출 산출물 로그 파일들은 이전 버전의 내부 작업 공간이었던 `draft`(예: `rules_v1_1/draft_v1_2/`) 디렉토리에 아카이빙하여 보관합니다.

## 2. 시스템 보호 구역 및 불가침 규칙 (System Protection)

옵시디언 볼트(Vault) 및 지식 베이스의 일관성을 유지하기 위해, 아래의 시스템 보호 구역과 불가침 규칙을 지정합니다.

1. **디렉토리 격리 보호:** `_system/`, `_agents/` 등 언더바(`_`)로 시작하는 디렉토리는 시스템 운영 명령, 템플릿, 에이전트 룰을 관리하는 **보호(Read-Only) 구역**입니다.
2. **접근 금지:** 에이전트는 사용자의 명시적이고 구체적인 승인이나 지시가 없는 한, 해당 보호 경로에 위치한 파일을 자동으로 분류(이동)하거나 임의로 내용을 수정해서는 안 됩니다.
3. **태그 기반 잠금 (Frontmatter Lock):** 개별 파일 단위로 본문 상단 Frontmatter(YAML)에 `agent_locked: true` 파라미터 혹은 `#system/locked` 태그가 명시된 경우, 디렉토리 위치와 무관하게 시스템 문건으로 취급하여 동일한 조치(수정/이동 금지)를 적용합니다.
4. **심볼릭 링크 보호:** 원본 문서를 가리키기 위해 루트 혹은 특정 경로에 둔 에이전트/룰 심볼릭 링크(예: 루트의 `AGENTS.md`)는 절대 삭제되거나 경로가 변경되어서는 안 됩니다. 추가적인 룰 심볼릭 링크가 존재하는 경우에도 동일한 보호 원칙을 적용합니다.
