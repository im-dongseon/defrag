# Search Protocol (v1.3.1)

## Overview
이 스킬은 `R2_search.md` 규칙에 정의된 3단계 검색 프로토콜을 에이전트가 엄격하게 따르도록 강제합니다. AI 에이전트는 무거운 전역 텍스트 검색으로 인한 토큰 낭비를 막기 위해 이 프로토콜을 따라야 합니다.

## When to Use
- 레포지토리 내에서 특정 파일, 정보, 또는 개념을 찾고자 할 때.
- `grep_search`와 같은 전역 텍스트 검색 도구를 실행하기 직전.
- 필요한 정보가 어디에 있는지 확신할 수 없을 때.

## 3단계 탐색 프로토콜 (Rigid Workflow)

검색은 반드시 아래의 순서대로 진행해야 합니다. 1단계와 2단계를 충분히 거치지 않고 바로 3단계로 넘어가는 것은 금지됩니다.

### 1단계: 인덱스 스캔 (최저 비용)
1. 사용자의 질문을 바탕으로 관련 도메인을 식별합니다. (예: `10-19` 행정, `20-29` 기술, `30-39` 프로젝트).
2. 해당 도메인의 최상위 인덱스 파일(예: `30_index.md`, `20_index.md`) **단 1개만** 로드합니다.
3. 인덱스를 읽고 디렉토리 구조와 하위 디렉토리 후보군을 파악합니다.

**Red Flag:** 서로 다른 여러 도메인의 인덱스 파일들을 동시에 로드하지 마세요. 가장 연관성 높은 도메인 하나에만 집중합니다.

### 2단계: 자율 경로 탐색 (비용 최적화)
인덱스 스캔만으로 정보가 부족한 경우:
1. `list_directory` 또는 `glob` 도구를 사용하여 1단계에서 타겟팅한 도메인이나 하위 디렉토리의 파일/폴더명 목록을 스캔합니다.
2. 파일명과 폴더명을 기반으로 탐색 범위를 좁힙니다. (예: `회의록/` 또는 `참고자료/` 폴더 안을 집중적으로 확인).
3. 가장 유력한 파일 **단 1개만**을 선택하여 내용을 확인합니다 (`read_file`).

**Red Flag:** 이 단계에서 본문 전체 검색을 돌리지 마세요. 철저히 파일명과 디렉토리명에 의존하여 탐색해야 합니다.

### 3단계: 전역 본문 검색 (최후 수단, 최고 비용)
1, 2단계를 통해서도 필요한 정보를 찾지 못한 경우에만 예외적으로 수행합니다:
1. 옵시디언 플러그인이 활성화되어 있다면 `obsidian search <키워드>`를 1순위로 사용합니다.
2. 불가능한 경우 `grep_search`와 같은 전역 텍스트 검색 도구를 2순위로 사용합니다.
3. 검색어는 최소화하고, 검색 대상 경로를 최대한 좁혀 전체 볼트(Vault)를 스캔하는 일을 피하십시오.

**Red Flag:** 다른 모든 방법을 실패하기 전까지 옵시디언 볼트 루트에서 광범위하게 `grep_search`를 실행하는 것은 절대 금지됩니다.

## Common Mistakes
- **grep으로 직행하기:** 질문을 받자마자 습관적으로 `grep_search`를 실행하는 행위.
- **디렉토리 통째로 읽기:** 특정 폴더 안의 파일들을 훑어본다며 여러 파일에 연속적으로 `read_file`을 호출하는 행위. 2단계를 통해 가장 유력한 파일 1개로 범위를 좁혀야 합니다.
